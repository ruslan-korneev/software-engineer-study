# Что такое контроль версий?

[prev: 17-debugging](../../02-python/17-debugging.md) | [next: 02-why-use-version-control](./02-why-use-version-control.md)
---

## Определение

**Контроль версий** (Version Control) — это система, которая записывает изменения в файл или набор файлов с течением времени, позволяя вернуться к определённой версии позже. Это как "машина времени" для вашего кода.

## Основные концепции

### Версия (Revision/Commit)
Каждое сохранённое состояние проекта называется **версией** или **коммитом**. Версия содержит:
- Снимок всех отслеживаемых файлов
- Метаданные (автор, дата, сообщение)
- Уникальный идентификатор

### Репозиторий (Repository)
**Репозиторий** — это хранилище, содержащее все файлы проекта и полную историю их изменений. Репозиторий может быть:
- **Локальным** — на вашем компьютере
- **Удалённым** — на сервере (GitHub, GitLab, Bitbucket)

### История изменений (History)
Полный журнал всех версий проекта. Позволяет:
- Видеть кто, когда и что изменил
- Вернуться к любой предыдущей версии
- Сравнивать различные версии

## Типы систем контроля версий

### 1. Локальные системы (Local VCS)
Простейший тип — база данных на локальном компьютере.

```
Рабочая директория → Локальная БД версий
```

**Пример:** RCS (Revision Control System)

**Проблема:** Невозможно работать в команде.

### 2. Централизованные системы (Centralized VCS)
Единый сервер хранит все версии, клиенты получают файлы оттуда.

```
        [Центральный сервер]
              ↑    ↑    ↑
              |    |    |
    Клиент A  Клиент B  Клиент C
```

**Примеры:** CVS, Subversion (SVN), Perforce

**Преимущества:**
- Все знают, кто чем занимается
- Администраторы контролируют доступ

**Недостатки:**
- Единая точка отказа
- Без сервера — нельзя работать

### 3. Распределённые системы (Distributed VCS)
Каждый клиент полностью копирует репозиторий, включая всю историю.

```
    [Сервер] ←→ [Полная копия]
        ↑           ↑
        |           |
[Полная копия] [Полная копия]
   Клиент A       Клиент B
```

**Примеры:** Git, Mercurial, Bazaar

**Преимущества:**
- Работа офлайн
- Полное резервное копирование
- Гибкие рабочие процессы

## Базовая терминология

| Термин | Описание |
|--------|----------|
| **Repository** | Хранилище проекта с историей |
| **Commit** | Зафиксированное состояние проекта |
| **Branch** | Независимая линия разработки |
| **Merge** | Объединение изменений из разных веток |
| **Clone** | Полная копия репозитория |
| **Push** | Отправка изменений на удалённый сервер |
| **Pull** | Получение изменений с удалённого сервера |
| **Checkout** | Переключение между версиями/ветками |

## Аналогия из жизни

Представьте, что вы пишете книгу:
- **Без контроля версий:** `книга_v1.docx`, `книга_v2_final.docx`, `книга_v2_final_FINAL.docx`
- **С контролем версий:** Одна папка, где система сама отслеживает все изменения

## Что отслеживает система контроля версий?

- **Текстовые файлы:** код, конфигурации, документация (.py, .js, .md, .json)
- **Бинарные файлы:** изображения, PDF (с ограничениями)
- **Структуру директорий:** создание, переименование, удаление папок

## Пример работы (концептуально)

```
День 1: Создали файл main.py           → Версия 1
День 2: Добавили функцию calculate()   → Версия 2
День 3: Исправили баг в calculate()    → Версия 3
День 4: Ой, всё сломали!               → Откат к Версии 2
```

## Резюме

Контроль версий — это фундаментальный инструмент разработки, который:
1. Сохраняет историю всех изменений
2. Позволяет откатываться к любой версии
3. Обеспечивает командную работу
4. Создаёт резервные копии проекта
5. Документирует процесс разработки

В современной разработке **Git** является стандартом де-факто для контроля версий.
